## ЗАМЕЧАНИЯ

**1. Нейминг**

- Названия методов вставки/удаления.

Придерживайся единообразия- либо get/place, либо getEntity/placeEntity.  
Используй стандартные названия. Если нужно что-то куда-то поместить, называй это "add" или "put" 
```java
public Entity get(Coordinates position)  <-- Почему не getEntity(...)?
public boolean placeEntity(Coordinates position, Entity entity)  <-- Почему не place(...)?

//ПРАВИЛЬНО:
public Entity get(Coordinates position)  
public boolean put(Coordinates position, Entity entity) 
```
Основной перевод слова "place" это "место", то есть это слово существительное.  
В названии методов стандартных библиотек это слово нигде не используется в значении "вставить".

- Избыточно. 

Слово "Snapshot" здесь ничего не добавляет к пониманию того, что делает метод.  
По сути, здесь это слово-паразит
```java
List<PositionedEntity> getPositionedEntitiesSnapshot()
List<Entity> getEntitiesSnapshot()

//ПРАВИЛЬНО:
List<PositionedEntity> getPositionedEntities()
List<Entity> getEntities()
```

- Избыточно. 

Мы и так понимаем, что метод поиска в классе поиска пути ищет именно путь, а не что-то иное.  
```java
public final class PathFinder {
  public static List<Coordinates> findPathToNearest(...) {...}
}

//ПРАВИЛЬНО:
public final class PathFinder {
  public static List<Coordinates> find(...) {...}
}
```

Уточнение, что ищем ближайшую цель- лишнее. Потому что а разве бывает иначе?

Возникает вопрос- раз есть метод, который ищет ближайший путь, 
то может быть есть метод, который ищет путь не к ближайшей цели?
Но такого метода нет, всегда ищем ближайший путь.

- Не делай методов с названием "пытаться"
```java
void tryReproduce(WorldMap map, Coordinates currentPosition, Random random)
```

Каждый метод это команда.  
Команда должна быть однозначной: беги, спи, прыгайНаОднойНоге.  

Условный пример: 
Не должно быть команды "беги или хотя бы попытайся"- потому что непонятно, как такой командой пользоваться.

Если метод должен бежать, то он должен бежать.  
Если что-то мешает ему бежать, метод должен кинуть исключение.

Если внутри метода есть условие "бежать если", то это условие нужно вынести из метода.

- Встретившись в коде, можно будет сделать ошибочный вывод, что переменная "map" относится к типу Map
```java
WorldMap map

//ПРАВИЛЬНО:
WorldMap worldMap
```
*Oracle Java code conventions, part."Naming conventions"*

**2. Нарушение конвенции кода**

Системно нарушается конвенция кода.  
Конструкторы ставятся в самый низ, хотя должны стоять первыми среди методов.  
Внутренние классы объявляются в самом верху, хотя должны объявляться в самом низу мастер класса.

**4. class WorldMap**

- Нарушение конвенции кода.  
Объявление внутренних классов должно идти в самом низу, а не в самом верху.

- Неправильная перегрузка методов
```java
public Entity get(Coordinates position) {
  return cells.get(position);
}

public Entity get(int x, int y) {
  return cells.get(new Coordinates(x, y));
}

//ПРАВИЛЬНО:
public Entity get(Coordinates position) {...}

public Entity get(int x, int y) {
  return get(new Coordinates(x, y));
}
```

- Никогда не возвращай null
```java
private final Map<Coordinates, Entity> cells = new HashMap<>();
  
public Entity get(Coordinates position) {
  return cells.get(position);  <-- может вернуть null
}
```
Возврат null повышает риск возникновения NullPointerException в программе.  
*Мартин, "Чистый код", гл.7.7-8*  
*Ютуб, Немчинский "Почему нельзя возвращать NULL?"*

- При всех операциях с участием координаты(добавить, выдать, удалить и т.д.), нужно проверять координату на корректность. 

Если координата некорректна(находится вне пределов карты), нужно бросать исключение.  
Сейчас, если у карты спросить, пустая ли ячейка с координатой (+100500, -100500), то карта скажет, что она пустая
```java
public boolean isEmpty(Coordinates position) {
  return !cells.containsKey(position);
}

//ПРАВИЛЬНО:
public boolean isEmpty(Coordinates position) {
  validate(coordinates);  <-- Бросает исключение, если координата находится вне пределов карты
  return !cells.containsKey(position);
}
```
А правильный ответ- ячейки с такой координатой в карте нет вообще.

Ближайшая аналогия- стандартные хранилища типа List и массива.  
При попытке обратитья к ним по несуществующему индексу, бросается исключение.

- Не делай возврат результата операций.

Во-первых, это нарушает правило разделения команд и запросов.  
Во-вторых, это приводит к тому, что проверка результата операции становится необязательным действием и можно забыть это сделать
```java
boolean placeEntity(Coordinates position, Entity entity)

//ПРАВИЛЬНО:
void placeEntity(Coordinates position, Entity entity)
```
Если по какой-то причине не удалось выполнить команду, в данном случае- поместить существо в карту, то метод должен бросить исключение.  
*Мартин, "Чистый код", гл.3, "Разделение команд и запросов"*

- Сообщения исключений делай не формальными, а информативными
```java
throw new IllegalArgumentException("Invalid map size");

//ПРАВИЛЬНО:
String message = "Invalid map size: %d, %d".formatted(width, height);
throw new IllegalArgumentException(message);
```

- Метод совершения хода в карте- нарушение SRP.

Теперь можно вызвать из карты метод хода и телепортировать зайца из одного края в другой минуя все правила игровой логистики
```java
Карта карта = new Карта(100, 100);
Заяц заяц = new Заяц();

карта.placeEntity(new Координата(0, 0), заяц);
карта.moveEntity(new Координата(0, 0), new Координата (99, 99));

/* class Карта */
public void moveEntity(Coordinates from, Coordinates to) {
  //17 строк логики совершения хода
}
```

Должна ли карта учитывать логистику зайцев? Если да, то как?

Метод *частично* пытается ответить на этот вопрос- он содержит 17 строк кода логики совершения хода.  
Но единственно правильный ответ- карта вообще не должна иметь в себе метод телепортации.  
Потому что в этом случае выполнение единого действия,- совершение хода существом,- разделяется на несколько классов.

+ 👍 В целом, с точки зрения SRP класс хороший.  
Имеющиеся нарушения SRP в нем- незначительные.

Если такой ситуации нет, то вынесение кода в утилиту является дроблением единой сущности и вынесением из нее части ответственности в другой класс.

**8. abstract class Creature extends Entity**

- Если объект пытаются создать с некорректными данными- бросай исключение
```java
public Creature(int hp, int maxHp, int speed) {
  this.maxHp = Math.max(1, maxHp);
  this.speed = speed;
  setHp(hp);
}

//ПРАВИЛЬНО:
public Creature(int hp, int maxHp, int speed) {
  //если hp и maxHp некорректны- кинуть исключение
  this.hp = hp;
  this.maxHp = maxHp;
  this.speed = speed;
}
```
Именно так ты и делаешь в Карте- бросаешь исключение из конструктора, если ширина или высота некорректные. 

+ 👍 Существо размножается, это прикольно. 

- Креатура должна сама искать свои текущие координаты
```java
public abstract void makeMove(WorldMap map, Coordinates position, Random random);

//ПРАВИЛЬНО:
public abstract void makeMove(WorldMap map, Random random);
```
Креатура все равно в метод хода принимает `WorldMap`, поэтому может спросить у карты свое текущее местоположение.  

Если креатура будет принимать свое текущее местоположение извне, а не получать самостоятельно из карты, то это потенциально может привести к багам. 
Потому что креатура не может гарантировать, что пришедшая извне координата *действительно* соответствует ее текущему местоположению.  
Слепая вера клиенту тут вредна.

- Также креатура не должна принимать Random
```java
public abstract void makeMove(WorldMap map, Coordinates position, Random random);

//ПРАВИЛЬНО:
public abstract void makeMove(WorldMap map);
```
Потому что класс Random это подробность внутренней организации класса Creature и/или его наследников.  
Свои внутренности класс должен создавать сам.

Передача Random в другие классы это артефакт процедурного мышления.  
Делается это обычно из соображений экономии ресурсов.  
Но такая экономия на спичках не оправдывает ухудшение ООП архитектуры.

+ 👍 С точки зрения SRP класс ок.

Например, сущность, которая управляет симуляцией, получая от юзера команды старт/стоп.  
Сущность, которая создает карту, etc.

- Божественный метод.

Ацки длинный метод main, который делает много разного- 74 строки!

Такие методы почти нечитаемы, а этот особенно- в нем сложный код.  
Делай маленьките методы, каждый из которых должен делать одно дело на одном уровне абстракции.  
*"ЧК", гл.3, "Правило одной операции", "Один уровень абстракции"*

**Совет:**  
Для тренировки и понимания некоторых ООП концепций, создай несколько майн-классов.  
Один пусть запускает игру так, как это сейчас- с конфигурированием карты через диалог с юзером.  
Второй майн пусть сразу запускает игру сразу с картой размером 10x10.
