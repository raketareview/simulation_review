**1. Нейминг**

- Названия классов нужно писать стилем PascalCase.

Названия не должны сбивать с толку. Трава "живая"- ее можно съесть и удалить с карты.  
Но она тут наследуется от Non_live_obj
```java
class Non_live_obj
class Grass extends PascalCase

//ПРАВИЛЬНО:
class НеходячийEntity
class Grass extends НеходячийEntity
```

- Названия классов нужно писать стилем PascalCase.  
В названиях не используй венгерскую нотацию
```java
public enum Enum_obj {
  Tree("🌲"),
  ...
}

//ПРАВИЛЬНО:
public enum EntityType
```

- Не используй в назывании пакетов слово "object".

Если пакет называется "object", это значит, что там храняться классы типа Object.  
Но это бессмыслено, потому что в java все классы неявно наследуются от класса Object,
поэтому вообще все классы относятся к типу Object и являются объектами
```java
package object;

//ПРАВИЛЬНО:
package entities;
```

- "Алгоритм А" это не то название, которое всем будет понятно без объяснения.  
Твои названия должны быть понятны не только тебе самому, но и твоим коллегам по команде

**5. Пустые блоки часто сигнализируют о неоптимальном алгоритме**
```java
public void makeMove() {
  if (attemptToEat()) {
  } else {
    movement();
  }
}

//ПРАВИЛЬНО:
public void makeMove() {
  if (!attemptToEat()) {
    movement();
  } 
}
```

**6. Приоритет- читаемость кода, а не экономия строк**

Не стремись склеить много инструкций в одну в ущерб читаемости кода.  
Здесь в одной строке выполняется 7 действий и это примерно абсолютно нечитаемо.  
Вводи поясняющие переменные
```java
int remainingHP = ((Creature) getSimProcess().getEntityLocations().get(target)).getHp() - powerAttack;
```

**7. Если в блоке if есть return(break, continue, throw, exit и т.д.), то else не пишется.**
 
В этом случае неважно, будет else или нет, так как программа будет работать одинаково, а код без `else` будет выглядеть читабельней
```java
if (targetAchieved(target)) {
  //...  
  return true;
} else {
  return false;
}

//ПРАВИЛЬНО:
if (targetAchieved(target)) {
  //...  
  return true;
} 
return false;
```

**8. Не используй ломающие мозг конструкции**

Многие зависнут на этой строчке и будут думать, что сдесь произойдет- сначала изменится значение поля, а потом оно распечатается, или наоборот?
```java
System.out.println("Ход № " + ++turnNumber);

//ПРАВИЛЬНО:
turnNumber++;
System.out.println("Ход № " + turnNumber);
```
Чтобы не путать, всегда используй постфиксную форму инкремента(i++), а не префиксную(++i).

**13. class Entity**

- Нарушение SRP.

Класс хранит разное, которое к сущности существа не имеет отношение.  
Например, процесс симуляции 
```java
abstract public class Entity {
  private final SimProcess simProcess;
  //...
}
```
Если Заяц хранит Симуляцию, значит между ними связь " has-a" и процесс симуляции является частью Зайца.  
Но должно быть ровно наорот- не зайцы должны хранить симуляцию, а симуляция в том или ином виде должна хранить зайцев.

Если копнуть, то окажется, что сейчас в проекте циклическая зависимость и спагетти код:  
Процесс симуляции хранит зайцев, а зайцы хранят процесс симуляции.  
Уши зайцев торчат отовсюду, этому нужно положить край.

- Нарушение SRP, зависимость модели от представления- существо хранит спрайт с собственным изображением
```java
private final Enum_obj entityType;
```
Модель(а это модель) не должна зависеть от представления и знать, как ее будут показывать юзеру.  
Потому что в разных средах(консоль, Swing, Android) одна и та же модель может быть показана разными способами- пиксельной картинкой, анимацией etc.  
Спрайты всех существ должны храниться в классе, который распечатывает карту.

- Содержит координату. Но координата нужна только тому существу, которое ходит.  
Поэтому entities должны хранить координату только начиная с уровня `Creature`
```java
private int x, y;
```
